**Keyword Expansion section** (KWRD-02):

Section heading: `## Keyword Expansion`

Expand the user's seed list using two methods:

1. **Apple Search Autocomplete** -- For each seed keyword (up to 10 seeds to avoid excessive web searches), use WebSearch to query: `{keyword} app site:apps.apple.com`. Extract related terms that appear in competitor titles and subtitles. Also try queries like `{keyword}` + common modifiers (best, free, top, for + use case). Label discovered terms as: `Source: Apple autocomplete [OBSERVED]`

2. **Claude Semantic Expansion** -- For each seed keyword, generate related terms across these dimensions:
   - Synonyms (e.g., "meditation" -> "mindfulness", "zen")
   - Long-tail variations (e.g., "meditation" -> "guided meditation for sleep", "morning meditation routine")
   - Feature-adjacent terms (e.g., "meditation" -> "breathing exercises", "sleep sounds")
   - Problem-solving phrases (e.g., "meditation" -> "reduce stress", "fall asleep faster")
   - Category terms (e.g., "meditation" -> "wellness app", "mental health")
   Label these as: `Source: semantic expansion [ESTIMATED]`

3. **Deduplication** -- Remove exact duplicates. Merge near-duplicates (singular/plural, minor spelling variants) keeping the more common form.

Target: Expand seed list to 30-50 total candidates (including originals). If seeds are already >30, expand by 50% rather than to a fixed target.

After expansion, confirm: "Expanded {seed_count} seeds to {total_count} keyword candidates ({expanded_count} new terms discovered)"

**Relevance Scoring section** (KWRD-03):

Section heading: `## Relevance Scoring`

Score every keyword (seed and expanded) for relevance to the app on a 1-10 scale:

| Score | Meaning |
|-------|---------|
| 9-10 | Core feature or primary use case of the app |
| 7-8 | Strongly related feature or common user need the app addresses |
| 5-6 | Moderately related -- relevant audience but not a direct feature |
| 3-4 | Tangentially related -- same broad category but weak fit |
| 1-2 | Irrelevant or misleading -- would attract wrong users |

For each keyword, provide: score, one-sentence explanation of semantic fit.

Base relevance on: app description (from .aso-context.json or user input), app category, title/subtitle keywords, and the keyword's relationship to what the app actually does.

Format as a table with columns: Keyword, Relevance, Explanation, Source.

Label the relevance score column as [ESTIMATED] in the section header since it is Claude's analytical assessment.

Sort the table by relevance score descending (highest relevance first).

**Data Labeling Convention section** -- same pattern as the final section of audit.md. Reference rules/aso-domain.md. Include the three label types: OBSERVED, ESTIMATED, and USER-PROVIDED (the new label for this command).

Target file length: 200-280 lines. This plan covers input resolution, expansion, and relevance scoring. Plan 02 will add user-data handling, popularity signals, and intent grouping.
