<objective>
Create the two app context utility commands: /aso:app-new (attaches an app by fetching and storing its metadata) and /aso:app-clear (removes stored context).

Purpose: These commands establish the "active app" pattern so users can run /aso:audit, /aso:keywords, and /aso:optimize without re-specifying the app each time. The .aso-context.json file they manage becomes the shared data contract across all /aso:* commands.

Output: `commands/aso/app-new.md` and `commands/aso/app-clear.md` -- two command files that will become /aso:app-new and /aso:app-clear slash commands.

Create `commands/aso/app-new.md` with the following structure:

**YAML Frontmatter (required fields):**

The file must start with a YAML frontmatter block containing exactly these fields:
- `name: aso:app-new`
- `description: Attach an app for ASO analysis. Fetches store listing metadata and saves to .aso-context.json. Run before /aso:audit for a streamlined workflow.`
- `allowed-tools:` list containing WebSearch, WebFetch, Read, Write, Bash
- `argument-hint: "<store-url-or-app-name>"`

**Command body must contain these sections in order:**

1. **Role Statement** (2-3 lines): You are the app context manager for the ASO Toolkit. Your job is to fetch an app's store listing metadata and save it as the active app context.

2. **Input Handling** (per ACTX-01 from CONTEXT.md):
   - Accept `$ARGUMENTS` as either a store URL or app name
   - Apply platform detection from rules/aso-domain.md:
     - `apps.apple.com/*` or `itunes.apple.com/*` = iOS
     - `play.google.com/store/apps/*` = Android
     - Numeric ID (9-10 digits) = iOS
     - Reverse-domain bundle ID (com.x.y) = Android
     - Plain text app name = search both stores, ask user to pick if multiple results
   - If no argument provided, ask user for a store URL or app name

3. **Data Fetching Strategy**:
   - **iOS apps with numeric ID**: Use WebFetch to call iTunes Search API: `https://itunes.apple.com/lookup?id={NUMERIC_ID}` -- returns JSON with trackName, description, averageUserRating, userRatingCount, genres
   - **iOS apps from URL**: Extract numeric ID from URL path (pattern: `/id(\d+)`), then call iTunes API
   - **Android apps**: Use WebSearch for `site:play.google.com/store/apps/details {app-name-or-url}`, extract metadata from results
   - **App name (no URL)**: Use WebSearch for `"{app name}" site:apps.apple.com OR site:play.google.com`, present results and ask user to confirm which app
   - **Fallback**: If iTunes API fails, fall back to WebSearch for the store listing page

4. **Context File Schema** -- Write `.aso-context.json` to the current working directory using the Write tool with this exact JSON structure containing 11 fields:
   - `app_name` (string): Display name of the app
   - `store_url` (string): Full store URL used for lookup
   - `platform` (string): "ios" or "android"
   - `title` (string): App title as shown in store
   - `subtitle` (string or null): iOS subtitle, null for Android
   - `short_description` (string or null): Google Play short description, null for iOS
   - `description` (string): Full description text
   - `rating` (number or null): Average star rating, e.g. 4.7
   - `rating_count` (number or null): Total number of ratings
   - `category` (string or null): Primary store category
   - `fetched_at` (string): ISO 8601 timestamp of when data was fetched

5. **Confirmation Output** (per CONTEXT.md specifics): After saving, print exactly:
   `Attached: {app_name} ({platform}) -- {rating} stars, {rating_count} ratings`
   If rating unavailable: `Attached: {app_name} ({platform}) -- rating unavailable`

6. **Error Handling**:
   - If store URL returns no results: "Could not find app at {url}. Check the URL and try again."
   - If app name search returns multiple results: Present numbered list, ask user to pick
   - If WebSearch fails entirely: "Web search unavailable. Try providing a direct store URL."

**File target: 80-120 lines.** Command is loaded only on invocation, so size is less constrained than rules.
