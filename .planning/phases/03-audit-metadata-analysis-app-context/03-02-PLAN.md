Create `commands/aso/audit.md` with the following structure. This is a large, detailed command file (200-350 lines) because it is loaded only on invocation and must guide Claude through a complex multi-step analysis.

**YAML Frontmatter (required fields):**

The file must start with a YAML frontmatter block containing exactly these fields:
- `name: aso:audit`
- `description: Audit an app store listing with detailed metadata analysis. Analyzes title, subtitle, description, and keyword strategy against platform-specific best practices.`
- `allowed-tools:` list containing WebSearch, WebFetch, Read, Write, Bash, Agent
- `argument-hint: "<store-url-or-app-name>"`

Key frontmatter decisions:
- `Agent` in allowed-tools prepares for Phase 4 (competitive analysis via aso-analyst subagent)
- No `model` field (inherit session model)
- No `disable-model-invocation` (allow Claude to auto-invoke when user says "audit my app")

**Command body must contain these sections in this exact order:**

**SECTION 1: Role and Scope** (5-8 lines)

You are the ASO Audit command. You perform a comprehensive analysis of an app's store listing metadata, evaluating each text field against platform-specific best practices defined in `rules/aso-domain.md`.

This audit covers metadata analysis: title, subtitle/short description, description text, and keyword field strategy. Scoring, competitive context, and report output are handled by subsequent phases and will be added to this command file later.

**SECTION 2: Input Resolution** (15-25 lines)

Handle three input paths in priority order:

Path A -- Explicit argument: If `$ARGUMENTS` is provided:
  - Apply platform detection per rules/aso-domain.md (URL patterns, numeric ID = iOS, bundle ID = Android, app name = search)
  - For URLs: extract app identifier, fetch metadata via WebSearch or iTunes API
  - For app names: WebSearch for `"{app name}" site:apps.apple.com OR site:play.google.com`, present results if ambiguous
  - Print: "Auditing: {app_name} ({platform})"

Path B -- Active app context: If no argument provided, check for `.aso-context.json` in the current working directory using the Read tool:
  - If file exists: load it, use stored metadata, print "Using active app: {app_name}" (per CONTEXT.md decision: silent use, no confirmation prompt)
  - If file does not exist: ask user for a store URL or app name

Path C -- No input: If neither argument nor context file, ask: "Provide an App Store or Google Play URL, or an app name to audit."

After input resolution, the command must have: platform (ios/android), title, subtitle or short_description, description text. Rating and category are used if available but not required for metadata analysis.

**SECTION 3: Data Fetching** (15-20 lines)

If metadata was loaded from .aso-context.json, skip fetching (data already available).

If fetching from a URL or app name:
- **iOS with numeric ID**: WebFetch `https://itunes.apple.com/lookup?id={ID}` for JSON data. Extract: trackName (title), description, averageUserRating, userRatingCount, genres. Note: iTunes API does not return subtitle -- use WebSearch for the App Store page to get subtitle.
- **iOS from URL**: Extract numeric ID from URL path (regex: `id(\d+)`), then call iTunes API as above.
- **Android**: WebSearch for the Google Play listing page. Extract: title, short description, full description, rating, rating count, category from page content.
- **App name (no URL)**: WebSearch `"{app name}" app store listing`, identify platform from results, extract metadata.

After fetching, confirm what was retrieved: "Retrieved metadata: title ({char_count} chars), subtitle ({char_count} chars), description ({char_count} chars)"

**SECTION 4: Title Analysis** (30-40 lines) -- Per CONTEXT.md: ordered first (highest ranking weight)

Analyze the app title against platform-specific rules from aso-domain.md.

Report these dimensions:
1. **Character Usage**: "{current_length}/{limit} characters used ({remaining} remaining)"
   - iOS limit: 30 chars. Google Play limit: 30 chars. Reference rules/aso-domain.md.
   - Flag if over limit (CRITICAL), if significantly under-utilized (>10 chars unused = HIGH opportunity)

2. **Keyword Presence**: Identify keywords in the title.
   - What keywords are present? List them.
   - Are high-value keywords front-loaded (appear in first half of title)? Front-loading increases ranking weight.
   - Are there obvious missing keywords for the app's category?

3. **Brand vs Keyword Balance**: Evaluate the title structure.
   - Pure brand name with no keywords: Flag as missed opportunity (unless globally recognizable brand like "Spotify" or "Netflix")
   - Keyword-stuffed with no brand: Flag as readability concern and potential policy risk
   - Balanced approach (e.g., "BrandName - Key Feature"): Note as good practice
   - For Google Play: check against prohibited words list from aso-domain.md (no "Best", "#1", "Free", "Top", ALL CAPS, emojis)

4. **Readability**: Is the title readable as a natural phrase or name? Keyword stuffing that creates nonsensical strings damages conversion.

Label all analysis as ESTIMATED (Claude's assessment). Label character counts as OBSERVED (measured from actual metadata).

**SECTION 5: Subtitle / Short Description Analysis** (25-35 lines) -- Per CONTEXT.md: ordered second

Platform-branched analysis:

**iOS Subtitle (30 chars):**
1. **Character Usage**: "{current_length}/30 characters used"
   - Flag significant underuse (>10 chars unused)
2. **Keyword Coverage**: What keywords appear in the subtitle?
   - Are they DIFFERENT from title keywords? Apple combines title + subtitle for indexing, so duplication wastes characters.
   - Flag any keywords that repeat from the title as "Wasted -- already indexed via title"
3. **Value Proposition**: Does the subtitle communicate what the app does or why users should care?
   - Strong: communicates benefit ("Track your sleep naturally")
   - Weak: generic descriptor ("The best app") or redundant with title

**Google Play Short Description (80 chars):**
1. **Character Usage**: "{current_length}/80 characters used"
   - 80 chars gives more room than iOS subtitle -- is it well-utilized?
2. **Keyword Coverage**: What search-relevant keywords are present?
   - Short description IS indexed on Google Play -- keyword presence matters for ranking
   - Are keywords naturally integrated or awkwardly stuffed?
3. **Value Proposition Clarity**: Does it communicate the core value in one scannable line?
   - This is the first text users see on the listing -- does it compel a tap?

If subtitle/short description is missing or empty, flag as CRITICAL: "No subtitle/short description found. This field directly impacts search ranking."

**SECTION 6: Description Analysis** (35-50 lines) -- Per CONTEXT.md: deep analysis, platform-branched

This section MUST branch by platform because iOS and Google Play descriptions serve fundamentally different purposes.

**iOS Description (4,000 chars, NOT indexed for search):**

Since Apple does NOT index the description for search, analyze for conversion quality:

1. **First 3 Lines Test**: The description is truncated after approximately 3 lines on the store page. Do the first 3 lines:
   - State the core value proposition clearly?
   - Create urgency or curiosity to read more?
   - Avoid wasting space on developer name or generic statements?

2. **Readability and Scannability**:
   - Are there short paragraphs or bullet points? (Wall of text = bad)
   - Is the language benefit-focused ("Track your sleep") vs feature-focused ("Sleep tracking feature")?
   - Estimated reading level: aim for grade 6-8 for broad accessibility

3. **Feature Coverage**: Does the description cover the app's key features?
   - List the features mentioned
   - Are they presented as user benefits or technical specs?

4. **Call to Action (CTA)**: Does the description end with a clear CTA?
   - Strong: "Download now and start sleeping better tonight"
   - Weak: just trails off with feature list
   - Missing: no CTA at all (flag as MEDIUM opportunity)

5. **Social Proof**: Are there mentions of awards, press coverage, user count, or ratings?
   - Present: note what social proof is used
   - Missing: flag as opportunity ("Consider adding social proof: user count, awards, press mentions")

**Google Play Description (4,000 chars, IS indexed for search):**

Since Google Play DOES index the full description, analyze for keyword integration AND readability:

1. **Keyword Density**: Count instances of likely target keywords in the description.
   - Benchmark: approximately 1 exact keyword match per 250 characters of description (ESTIMATED guideline from ASO industry)
   - Too few: missing ranking opportunity
   - Too many: keyword stuffing risk (Google may penalize)
   - Calculate: "{keyword_count} keyword instances in {char_count} chars = 1 per {chars_per_keyword} chars"

2. **Front-Loading**: Are the most important keywords in the first 1-2 paragraphs?
   - Google weights early description content more heavily
   - First paragraph should contain the primary target keywords naturally

3. **Scannability**:
   - Short paragraphs (3-4 sentences max)?
   - Bullet points or feature lists?
   - Headers or emoji separators for visual structure?

4. **Natural Language Flow**: Do keywords read naturally in sentences, or are they awkwardly inserted?
   - Natural: "Track your daily steps and monitor your fitness goals"
   - Stuffed: "step counter fitness tracker pedometer walking app exercise"

5. **First 3 Lines + CTA**: Same conversion analysis as iOS (truncation, value prop, CTA)

**SECTION 7: Keyword Field Guidance -- iOS Only** (20-30 lines) -- Per CONTEXT.md: strategy advice only (cannot observe actual field)

This section appears ONLY for iOS apps. For Android apps, skip this section entirely (Google Play has no hidden keyword field).

Per CONTEXT.md locked decision: This is guidance only. The actual keyword field contents cannot be observed from outside the app's App Store Connect account. Frame all advice as strategic recommendations.

Provide guidance on the 100-character keyword field strategy:

1. **Formatting Rules Reminder** (reference aso-domain.md):
   - Comma-separated, NO spaces after commas (spaces waste characters)
   - Do NOT repeat words already in the Title or Subtitle (Apple auto-combines all three for indexing)
   - Prefer singular forms (Apple indexes both singular and plural automatically)
   - Do not use competitor brand names (policy violation risk)
   - Avoid prepositions and articles (waste space)

2. **Coverage Strategy**:
   - Identify keywords that are NOT already in the title or subtitle
   - Suggest high-value keywords from the app's category and feature set
   - Estimate how many unique terms can fit in 100 chars (typically 15-25 short keywords)
   - Provide an example keyword string format: `keyword1,keyword2,keyword3,...`

3. **Opportunity Assessment**:
   - Based on the app's title, subtitle, and description, identify keyword gaps
   - Suggest the top 5-8 keywords that should be in the keyword field
   - Frame as: "Consider including: {keyword1}, {keyword2}, ..." (since actual field is unobservable)

Label all keyword suggestions as ESTIMATED. Note explicitly: "Keyword field contents cannot be observed externally. These are strategic recommendations."

**SECTION 8: Analysis Summary** (10-15 lines)

After all analysis sections, provide a brief summary:

1. **Quick Stats Table**:

    | Field | Length | Limit | Usage |
    |-------|--------|-------|-------|
    | Title | {n} | {limit} | {percent}% |
    | Subtitle/Short Desc | {n} | {limit} | {percent}% |
    | Description | {n} | 4,000 | {percent}% |

2. **Top Findings**: List the 3 most impactful findings from the analysis (highest priority items to fix).

3. **Next Steps Note**: "For scoring, competitive context, and a full report file, these features will be available in a future update." (Phases 4 and 5 will add scoring and output.)

**SECTION 9: Data Labeling Reminder** (3-5 lines)

Remind Claude to follow the OBSERVED/ESTIMATED labeling convention from rules/aso-domain.md throughout the output:
- Character counts, ratings, review counts = OBSERVED
- Keyword assessments, quality evaluations, strategic recommendations = ESTIMATED
- When in doubt, label as ESTIMATED (safer default)

**Total file target: 200-350 lines.** The audit command is the most complex command in the toolkit. It is loaded only when invoked, so the line budget is generous.
